# Feature Notes

## 8. Scale Backdrop / Neck Context (formerly "CAGED Scales")

CAGED as a formal concept does not generalize beyond Guitar.STANDARD and does not
need to be implemented as a named system. The useful application is a scale backdrop
on the neck visualization: given the current playhead position, overlay all valid
scale note positions within a fret window as dim dots on the neck.

At any point in playback the system already knows:
- Current (fret, string) from the playhead
- Key and mode from the lick

The backdrop is findNeckPositions called for every interval in the mode's sequence,
filtered to a fret window around the current position, rendered as dim markers.
Current note is bright, valid scale notes in window are dim, everything else empty.

This is Guitar-specific by nature since it is a visual neck display. No formal CAGED
window system needs to be defined or named in code. The fret-range filtering is already
how spanLimit works in the builders — no new infrastructure required.

Answers the practical player question: "I am here on the neck, what other notes are
available in this mode?"

Depends on: #19 (neck visualization).


## 11. Parallel Harmony

Static phrase transformation. Operates entirely in interval-space; never touches
fingering until the final position pipeline call.

Implementation:
- Input: List<IntervalNote> (a stored lick's intervals), Mode, harmonicInterval
  (e.g. THIRD, SIXTH)
- Map each IntervalNote to its diatonic equivalent N scale degrees up within the mode's
  interval sequence. This is a pure index shift on the mode's degree list, wrapping
  at the octave.
- Output: new List<IntervalNote> — feed directly into existing PositionBuilder pipeline
- Add a snap-to-scale flag for handling chromatic/passing tones in the input: when true,
  round the input note to the nearest scale degree before harmonizing; when false,
  preserve the chromatic interval exactly.
- Lives in LickUtils as a stateless pure function alongside toIntervals and detectMode.


## 15. Phrase Predictor

Deterministic rule engine. No AI or probabilistic components. Operates in interval-space
over a chord progression input and emits an interval sequence fed through the existing
position pipeline.

Rules priority hierarchy (implemented as weighted candidate scoring):
1. Chord tones on strong beats — target 1, 3, 5 (and 7) of the current chord
2. Stepwise voice leading — prefer ±1-2 scale degrees between consecutive notes
3. Tension/resolution directionality — certain degrees resolve in defined directions
   per mode
4. Passing tones on weak beats — non-chord tones permitted between strong beat targets

Mode intervals are native — compute in whatever mode is active. Do not convert to
relative Ionian internally. Aeolian computes as 1 2 b3 4 5 b6 b7 from its own root.
Relative Ionian equivalence is a display label only.

Input: key, Mode, chord progression (sequence of chord root + type mapped to intervals),
optional seed phrase as List<IntervalNote>.
Output: List<IntervalNote> — renderable via existing position pipeline.

Guitar.STANDARD may attach a weighted hint table of idiomatic interval jumps (see #20)
as priority candidates for step 2 without affecting the instrument-agnostic rule core.

Lives in LickUtils or a dedicated PredictorService. No instrument knowledge.


## 16. Playback Tool & Song DB

Two decoupled concerns — build playback first, add the data layer separately.

Metronome:
- Web Audio API AudioWorklet for drift-free timing (not setInterval)
- Tempo (BPM) and time signature selection
- Visual pulse tied to the audio clock

Song DB:
- Table: song(id, title, artist, key, tempo, mode, vocal_range?)
- Decoupled from playback implementation — add after playback is stable
- Ties into chord sheet display as a separate feature layer


## 17. Octave Tracking

No string gauge attribute needed on Instrument. Derive absolute pitch from
fret + string using open string offsets, then add semitones per fret.

Standard tuning absolute pitches:
  String 0 (low E) = E2, String 1 = A2, String 2 = D3,
  String 3 = G3,   String 4 = B3,  String 5 = E4
Absolute pitch = openStringPitch[stringIndex] + fret

Add octave as a derived field on TabNote or a wrapper — computed at render time from
the Instrument's open string absolute pitches, not stored. Each Instrument implementation
provides its open string absolute pitches alongside existing tuning(). Gauge affects
tone not pitch and is out of scope.


## 18. Microphone Pitch Detection

Implement as a separate Fretboard Explorer mode, not integrated into tab playback
initially. Tab-following requires timing sync on top of pitch detection and is a
later phase.

Implementation:
- YIN algorithm or pitchfinder.js for single-note pitch detection
- Limit to Guitar.STANDARD, clean tone input for accuracy
- On note detected: resolve to nearest Note enum value, display all neck positions
  for that note via findNeckPositions
- This is the reverse of tab display — user plays, app shows where it lives on the neck
- Distortion and low notes degrade detection accuracy; document as known constraint
- The existing upload pipeline hardcodes Guitar.STANDARD in parseTab + toIntervals;
  resolve this before building tab-following on top of the audio pipeline


## 19. Animated Neck Visualization

Playhead-driven neck display. Notes flash on active fret/string positions as the
tab plays at the current tempo.

Implementation:
- Playhead indexes into the tab's columnIndex sequence at a rate derived from BPM
- At each columnIndex advance, highlight the corresponding TabNote positions on the neck
- Pairs directly with the metronome from #16 — tempo is the shared clock
- Instrument-aware: render string count and layout from the Instrument interface,
  not hardcoded to 6 strings
- Scale backdrop (#8) overlays dim dots for valid mode positions in the current fret
  window — Guitar only


## 20. Idiomatic Neck Traversal Hints

Physical jump patterns from CAGED literature are Guitar.STANDARD-specific and attach
there. These are interval relationships with idiomatic physical expressions in standard
tuning, not universal geometry. Do not add to the Instrument interface.

Implementation:
- Define as a static structure on Guitar.STANDARD:
  List of (intervalDelta, stringsTraversed) pairs representing common idiomatic moves
- Consumed by the predictor (#15) as a weighted hint table for interval selection —
  preferred candidate moves, not constraints
- The position pipeline already handles geometry: findCandidates locates nearby
  positions given current (fret, string) and a target interval. The hint table only
  influences which interval to target, not how to find it.
- Other instruments may define their own idiomatic jump tables attached to their own
  classes if desired, but there is no shared contract for this.
